# BVH to R2 Pipeline

This document extends the repository's manifest-first ingestion architecture for
serving CMU BVH motions as browser-friendly preview assets.

The existing pipeline remains the source of truth for motion identity and base
metadata:

```text
parse_motion_index.py  -> data/manifests/motion_index.json
parse_bvh_metadata.py  -> data/manifests/bvh_metadata.json
build_manifest.py      -> data/manifests/motions.json
import_postgres.py     -> PostgreSQL rows
```

R2 asset generation should be an additional offline stage after
`data/manifests/motions.json` exists.

## Target Pipeline

```text
Original BVH files
    |
Existing metadata pipeline
    |
data/manifests/motions.json
    |
Offline asset conversion
    |-- source BVH upload
    |-- GLB animation preview
    |-- WebP thumbnail
    |
data/manifests/r2_assets.json
    |
R2 upload / verification
    |
PostgreSQL asset-key import
```

The conversion stage should produce a deterministic asset manifest before the
database is updated. This keeps Blender conversion, object storage, and database
imports retryable as separate steps.

## R2 Object Layout

Use deterministic keys derived from `source_id`, not only from the filename.
Current CMU filenames are expected to be unique, but `source_id` is the canonical
motion identity in PostgreSQL.

For `source_id = cmu:01:01` and `filename = 01_01.bvh`:

```text
cmu/
  source/cmu_01_01.bvh
  previews/cmu_01_01.glb
  thumbnails/cmu_01_01.webp
```

Recommended object metadata:

```text
source_id: cmu:01:01
source_sha256: <BVH sha256 from motions.json>
asset_kind: source | glb_preview | thumbnail
conversion_version: <stable converter version>
```

The source BVH upload is optional for preview serving, but useful for provenance
and reproducibility.

## Asset Manifest

Generate `data/manifests/r2_assets.json` as a JSON array keyed by `source_id`.
Each record should include all R2 object keys and enough integrity data to retry
or verify uploads without re-running Blender.

Example:

```json
{
  "source_id": "cmu:01:01",
  "filename": "01_01.bvh",
  "relative_path": "001/01_01.bvh",
  "source_sha256": "abc123...",
  "conversion_status": "converted",
  "conversion_version": "bvh-glb-v1",
  "source_object_key": "cmu/source/cmu_01_01.bvh",
  "glb_object_key": "cmu/previews/cmu_01_01.glb",
  "thumbnail_object_key": "cmu/thumbnails/cmu_01_01.webp",
  "glb_sha256": "def456...",
  "thumbnail_sha256": "789abc...",
  "glb_size_bytes": 123456,
  "thumbnail_size_bytes": 12345,
  "thumbnail_width": 512,
  "thumbnail_height": 512,
  "error_message": null
}
```

Suggested `conversion_status` values:

```text
pending
converted
uploaded
skipped_invalid_bvh
conversion_failed
upload_failed
```

The existing joined motion manifest already carries `validation_status`. Records
with `validation_status = invalid` should not be sent to Blender. They should be
written to the asset manifest with `conversion_status = skipped_invalid_bvh` so
the batch result is complete and auditable.

## PostgreSQL Shape

The current `public.motions` table stores motion metadata only. To support the
catalog and viewer, it needs either asset columns on `public.motions` or a
separate `public.motion_assets` table.

A separate table keeps conversion state isolated from base motion metadata:

```sql
CREATE TABLE IF NOT EXISTS public.motion_assets (
    source_id TEXT PRIMARY KEY REFERENCES public.motions(source_id),
    conversion_status TEXT NOT NULL,
    conversion_version TEXT NOT NULL,
    source_object_key TEXT,
    glb_object_key TEXT,
    thumbnail_object_key TEXT,
    glb_sha256 TEXT,
    thumbnail_sha256 TEXT,
    glb_size_bytes BIGINT,
    thumbnail_size_bytes BIGINT,
    thumbnail_width INTEGER,
    thumbnail_height INTEGER,
    error_message TEXT,
    updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

The catalog backend can then query `motions` joined to `motion_assets` and return
signed or public R2 URLs for `thumbnail_object_key` and `glb_object_key`.

## Headless Blender Conversion

Run Blender only during ingestion or asset refresh:

```bash
blender --background \
  --python scripts/convert_bvh.py \
  -- --input input.bvh --glb output.glb --thumbnail output.webp
```

The script should coordinate only command-line parsing and file paths. Reusable
conversion helpers should live under `src/cmu_mocap_ingestor/`, matching the
project's existing development pattern.

The converter should:

1. Load one joined manifest record and its source BVH.
2. Skip records whose `validation_status` is not `valid`.
3. Import the BVH into Blender.
4. Validate that an armature and action were created.
5. Set the scene frame range from `frame_count`.
6. Set FPS from `frame_time` / `frame_rate`.
7. Preserve root translation and joint rotations.
8. Apply a documented scale and axis convention.
9. Export a binary glTF file.
10. Render a representative WebP thumbnail frame.
11. Calculate hashes and byte sizes for generated assets.
12. Write or update the asset manifest.

Important conversion decisions must be fixed in the manifest's
`conversion_version`. If scale, axes, thumbnail style, or export settings change,
increment the version so stale assets can be detected.

## GLB Preview Contract

A GLB preview should contain:

- the imported skeleton hierarchy
- animation keyframes
- root translations
- joint rotations
- a visible representation for preview rendering

A skeleton-only GLB may be technically valid, but it is not automatically visible
in Three.js without viewer-side helper geometry. Choose one of these contracts:

1. Export a simple skinned or stick-figure mesh in the GLB.
2. Export skeleton and animation only, and require the viewer to render a
   `SkeletonHelper` or equivalent debug geometry.

The thumbnail renderer and browser viewer must use the same visual contract so
catalog thumbnails resemble the loaded preview.

## Catalog Behavior

```text
Catalog page
    -> queries PostgreSQL for motion metadata and thumbnail object keys
    -> loads one page of WebP thumbnails from R2

User selects a card
    -> backend returns or signs the GLB object URL

Viewer
    -> loads the GLB with GLTFLoader
    -> plays its embedded animation
```

The browser should not parse BVH files during normal preview. BVH parsing and
Euler-to-animation conversion should happen once during ingestion.

## Retry Behavior

Keep conversion, upload, and database import independently retryable:

- If Blender conversion fails, record `conversion_failed` and the error message.
- If conversion succeeds but upload fails, keep local generated files and record
  `upload_failed`.
- If upload succeeds but database import fails, re-run only the asset-key import.
- If a source BVH hash changes, regenerate all derived assets for that `source_id`.
- If `conversion_version` changes, regenerate assets whose manifest version is
  older.

The database should reflect the latest asset manifest, not drive conversion
state directly.
