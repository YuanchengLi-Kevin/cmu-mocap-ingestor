# BVH to R2 Pipeline

```text
Original BVH
    ↓
One-time offline conversion
    ├── GLB animation preview
    └── WebP thumbnail
    ↓
Upload assets to R2
    ↓
Store metadata and object keys in PostgreSQL
```

## For each animation, R2 contains

```text
cmu/
├── source/01_01.bvh
├── previews/01_01.glb
└── thumbnails/01_01.webp
```

## Why GLB

Why GLB

A GLB can contain:

- the skeleton hierarchy
- binary animation keyframes
- root translations
- joint rotations
- optionally a simple mesh
- all necessary data in one file

Three.js loads GLB through GLTFLoader and returns its animations directly as AnimationClip objects. This avoids parsing BVH channel declarations and converting Euler rotations during every browser session.

## Catalog behavior

```text
Catalog page
    → loads 24 WebP thumbnails

User selects a card
    → loads one GLB from R2

Viewer
    → plays its embedded animation
```

## Headless Blender Conversion

```Bash
blender --background \
  --python scripts/convert_bvh.py \
  -- input.bvh output.glb
```

The conversion script would:

Import the BVH.
Validate the armature and animation.
Configure the export frame range.
Export a binary glTF file.
Render a representative frame to WebP.
Calculate hashes and sizes.
Upload both assets to R2.
Insert their object keys into PostgreSQL.

You run this once during ingestion, not every time someone previews an animation.
